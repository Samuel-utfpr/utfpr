Sistemas Operacionais

Capítulo 1: Introdução aos Sistemas Operacionais

1.1. O que é um Sistema Operacional?

Um Sistema Operacional (SO) pode ser definido como uma camada de software intermediária que se posiciona entre o hardware físico do computador e os programas aplicativos utilizados pelo usuário. Ele funciona como um grande gerente de recursos e um facilitador, cuja principal função é tornar o computador mais fácil de usar, além de gerenciar todos os seus recursos de forma eficiente, justa e segura.

A estrutura de um SO é inerentemente complexa. Em seu nível mais baixo, encontramos os drivers, que são softwares especializados em traduzir comandos genéricos (como "ler dados") em sinais elétricos específicos que um determinado dispositivo de hardware (uma placa de rede, um disco rígido) consegue entender. Em seu nível mais alto, temos os utilitários (como formatadores de disco, gerenciadores de arquivos) e a interface gráfica (GUI), que é a camada com a qual o usuário interage diretamente.

[Imagem de um diagrama mostrando o SO entre o hardware e os aplicativos]

O SO atua como uma ponte que abstrai a complexidade. Quando um usuário utiliza um editor de textos e salva um documento, ele não precisa se preocupar em como os bits e bytes daquele texto são fisicamente organizados e gravados nos setores de um disco magnético ou nas células de memória de um SSD. O SO lida com toda essa complexidade, oferecendo ao usuário e ao aplicativo uma interface muito mais simples e intuitiva: o conceito de "arquivo".

1.2. Funções Essenciais: Abstração e Gerência de Recursos

As duas funcionalidades primordiais de um sistema operacional são:

  • Abstração de Recursos: O SO esconde os detalhes e a complexidade do hardware, provendo aos aplicativos interfaces de alto nível, simples e homogêneas. Um desenvolvedor de software não precisa escrever um código específico para cada modelo de placa de som ou tipo de impressora existente. Ele simplesmente usa a API (Application Programming Interface) do SO para "tocar um som" ou "imprimir um documento". O SO, através de seus drivers, se encarrega de traduzir essa requisição para o hardware específico presente na máquina. Isso torna os aplicativos independentes do hardware.
  • Gerência de Recursos: O SO gerencia e coordena o uso de todos os recursos do computador — a Unidade Central de Processamento (CPU), a memória RAM, os dispositivos de Entrada/Saída (E/S) — pelos diversos programas que estão em execução. Isso inclui:
    • Compartilhamento do Processador: Em um sistema moderno, múltiplos programas parecem rodar ao mesmo tempo. O SO é quem orquestra essa ilusão, dividindo o tempo do processador entre os diversos programas de forma tão rápida que o usuário tem a impressão de que eles executam em paralelo.
    • Acesso Sequenciado: Para recursos que, por sua natureza, não podem ser usados por mais de um programa ao mesmo tempo (como uma impressora), o SO cria uma fila (spool) e garante que os trabalhos sejam executados um de cada vez, de forma ordenada.
    • Proteção: Uma das funções mais críticas do SO é proteger os recursos. Ele impede que um programa mal-comportado ou com falhas acesse a área de memória de outro programa, o que poderia corromper seus dados e causar instabilidade. Ele também evita que um único programa monopolize todos os recursos (um ataque de negação de serviço), garantindo a estabilidade e a segurança do sistema como um todo.

1.3. Áreas de Gerência

A gerência de recursos se desdobra em várias áreas específicas:

  • Gerência do Processador: Decide qual programa (processo) deve usar a CPU, em que momento e por quanto tempo. Essa tarefa é realizada por um componente chamado escalonador de processos.
  • Gerência de Memória: Aloca porções da memória principal (RAM) para os processos em execução e garante que cada um permaneça em sua área designada, criando espaços de memória isolados e protegidos.
  • Gerência de Dispositivos: Configura e controla o acesso a todos os periféricos, como teclado, mouse, monitores, discos e impressoras, através dos drivers.
  • Gerência de Arquivos: Organiza os dados em disco, criando e mantendo a estrutura de arquivos e diretórios, e controla as permissões de acesso a esses dados.
  • Gerência de Proteção: Define e implementa regras de acesso aos recursos, determinando quem pode fazer o quê (por exemplo, quais usuários podem ler ou modificar um arquivo).

1.4. Tipos de Sistemas Operacionais

Os sistemas operacionais são categorizados conforme sua aplicação e arquitetura:

  • Sistemas em Batch: Os primeiros sistemas operacionais. Executavam "lotes" de tarefas (jobs) sequencialmente, sem qualquer interação com o usuário durante o processamento. Eram comuns em mainframes para processar folhas de pagamento, transações bancárias, etc.
  • Sistemas Multiusuário: Permitem que múltiplos usuários acessem os recursos do computador simultaneamente. O SO é responsável por gerenciar as permissões, garantindo que um usuário não interfira no trabalho do outro (ex: Linux, Windows Server).
  • Sistemas de Desktop: O tipo mais comum para computadores pessoais. São focados na interatividade com um único usuário, com forte suporte a interfaces gráficas amigáveis (ex: Windows 11, macOS, distribuições Linux como o Ubuntu).
  • Sistemas Móveis: Otimizados para dispositivos portáteis como smartphones e tablets. Possuem características específicas como gerenciamento agressivo de energia, foco em conectividade (Wi-Fi, 4G/5G) e integração com sensores (GPS, acelerômetro) (ex: Android, iOS).
  • Sistemas Embarcados (Embarcados): Projetados para funcionar em hardware específico e com recursos limitados (pouca memória, processador menos potente). Estão presentes em inúmeros dispositivos do dia a dia, como em micro-ondas, sistemas de injeção eletrônica de carros, smart TVs, etc.
  • Sistemas de Tempo Real (Real-Time): Sistemas onde o tempo de resposta a um evento não é apenas uma questão de desempenho, mas de correção. São cruciais em sistemas críticos.
    • Hard Real-Time: Atrasos na resposta não são tolerados sob nenhuma circunstância, pois podem levar a falhas catastróficas (ex: sistemas de freios ABS de um carro, controle de voo de um avião, reatores nucleares).
    • Soft Real-Time: Atrasos são tolerados, embora não sejam desejados, pois degradam a qualidade do serviço, mas não causam uma falha crítica (ex: sistemas de streaming de vídeo, onde um atraso pode causar um "engasgo" na imagem, ou transações bancárias online).

1.5. Breve Histórico dos Sistemas Operacionais

  • Anos 40-50: A computação era "nua". Programas eram inseridos diretamente no hardware e executavam sozinhos. Com o tempo, surgiram as "bibliotecas de sistema" para reutilizar código de operações comuns de E/S.
  • Anos 60: A revolução do time-sharing (tempo compartilhado). Sistemas como o CTSS e o ambicioso Multics permitiram que múltiplos usuários interagissem com um único e caro computador mainframe simultaneamente. No final da década, frustrados com a complexidade do Multics, Ken Thompson e Dennis Ritchie, da Bell Labs, criaram uma versão mais simples e elegante: o UNIX.
  • Anos 80: A era dos computadores pessoais (PCs). A Microsoft lança o MS-DOS (1981), que dominou o mercado de PCs da IBM. A Apple lança o Mac OS 1.0 (1984) com o Macintosh, popularizando a interface gráfica do usuário (GUI) e o mouse. A Microsoft segue o caminho com o Windows 1.0 (1985).
  • Anos 90: Em 1991, um estudante finlandês chamado Linus Torvalds lança a primeira versão do kernel Linux. Combinado com as ferramentas do projeto GNU, o Linux se torna um sistema operacional completo, livre e de código aberto, que viria a dominar o mundo dos servidores. A Microsoft lança o Windows NT (1993), uma base de 32 bits muito mais robusta e segura que serviria de alicerce para todas as versões futuras do Windows.
  • Anos 2000-Hoje: A era da mobilidade e da nuvem. A Apple redefine o mercado com o iOS (originalmente iPhone OS) em 2007. No mesmo ano, o Google lança o Android, baseado no kernel Linux, que se torna o sistema operacional móvel mais popular do mundo. A virtualização (com softwares como VMWare) e a computação em nuvem se tornam tecnologias predominantes.

1.6. Estrutura de um Sistema Operacional

A forma como os componentes de um SO são organizados define sua arquitetura.

  • Núcleo (Kernel): O coração do SO. É o software que executa em modo privilegiado (nível núcleo), tendo acesso direto ao hardware. É responsável pela gerência de processos, memória e dispositivos.
  • Drivers: Módulos de software que atuam como tradutores, permitindo que o núcleo se comunique com os mais variados dispositivos de hardware.
  • Utilitários e Aplicativos: Todos os outros programas, que rodam em "modo usuário" e interagem com o hardware através dos serviços (syscalls) oferecidos pelo núcleo.

As principais filosofias de arquitetura para o núcleo são:

  • Sistemas Monolíticos: Todo o núcleo (gerência de processos, memória, sistema de arquivos, drivers de dispositivo, protocolos de rede) é compilado e executado como um único e grande programa em modo privilegiado. Vantagens: O desempenho é muito alto. Desvantagens: É uma arquitetura complexa de manter e depurar, e uma falha em um componente pode comprometer a estabilidade de todo o sistema. Exemplos: Linux, MS-DOS.
  • Sistemas Micronúcleo (Microkernel): A filosofia aqui é manter o núcleo o menor e mais simples possível, com outros serviços rodando como processos separados em modo usuário. Vantagens: Maior estabilidade e modularidade. Desvantagens: O desempenho é consideravelmente inferior devido à sobrecarga de comunicação. Exemplo: Minix 3.
  • Sistemas Híbridos: Tentam obter o melhor dos dois mundos. A maioria dos sistemas operacionais modernos, como Windows, macOS e até mesmo o Linux, são considerados híbridos.

1.7. Hardware Essencial e Níveis de Privilégio

Para que um SO possa gerenciar e proteger os recursos de forma eficaz, o hardware da CPU precisa oferecer suporte a, no mínimo, dois modos de operação, conhecidos como níveis de privilégio:

  • Nível Núcleo (Kernel Mode, Supervisor Mode, Ring 0): Neste modo, o software tem acesso total e irrestrito a todos os recursos do hardware. Ele pode executar qualquer instrução da CPU, acessar qualquer registrador e qualquer endereço de memória. Apenas o núcleo do SO e seus drivers rodam neste nível de altíssimo privilégio.
  • Nível Usuário (User Mode, User-space): Neste modo, o acesso aos recursos é rigidamente controlado e restrito pelo hardware. Um programa em modo usuário não pode executar instruções sensíveis (como desabilitar interrupções), não pode acessar diretamente dispositivos de hardware e não pode acessar áreas de memória que não lhe pertençam. Qualquer tentativa de realizar uma operação ilegal gera uma "exceção" de hardware, que força uma transição imediata para o modo núcleo, devolvendo o controle ao SO para que ele possa lidar com a infração (geralmente, finalizando o programa ofensor). Todos os aplicativos e utilitários rodam neste nível.

Essa separação é a base da proteção em sistemas operacionais modernos e é crucial para garantir a estabilidade do sistema.

1.8. Chamadas de Sistema (Syscalls)

Se um programa em nível de usuário tem acesso restrito, como ele pode realizar tarefas úteis como abrir um arquivo, alocar memória ou enviar dados pela rede, que exigem acesso a recursos controlados pelo núcleo? A resposta é através das chamadas de sistema (syscalls).

Uma syscall é o mecanismo formal e controlado através do qual um programa em modo usuário pode solicitar um serviço ao núcleo do SO. É a única "porta de entrada" legal para o modo núcleo. O processo funciona da seguinte forma:

  1. A aplicação chama uma função de uma biblioteca padrão (como write() da libc).
  2. Essa função de biblioteca prepara os parâmetros para a chamada de sistema, colocando-os em registradores específicos da CPU (ex: o número da syscall no registrador rax).
  3. A biblioteca então executa uma instrução especial de trap (como a instrução syscall em processadores x86-64).
  4. Essa instrução causa uma interrupção de software, que muda a CPU do modo usuário para o modo núcleo e transfere o controle para uma rotina de tratamento de syscalls do SO.
  5. A rotina do núcleo executa a operação solicitada (ex: sys_write).
  6. Após a conclusão, o núcleo coloca o valor de retorno em um registrador, executa uma instrução de retorno de interrupção (como sysexit), que faz a CPU voltar para o modo usuário, e devolve o controle à função da biblioteca.
  7. A função da biblioteca retorna o resultado para a aplicação.

Esse mecanismo garante que, embora o núcleo execute tarefas em nome do usuário, ele mantém o controle total sobre o sistema.

Capítulo 2: Processos e Escalonamento

2.1. O Conceito de Processo

O conceito de processo é um dos pilares de qualquer sistema operacional moderno. Um processo é, de forma simples, um programa em execução. É importante distinguir entre "programa" e "processo":

  • Um programa é uma entidade estática e passiva, um conjunto de instruções e dados armazenado em um arquivo em disco (um executável).
  • Um processo é uma entidade dinâmica e ativa, uma instância de um programa que está sendo executada. Ele tem um estado que muda ao longo do tempo.

Cada processo possui seu próprio espaço de endereçamento virtual, que é sua visão privada e isolada da memória. Nele, o processo armazena seu código executável (segmento de texto), suas variáveis globais e estáticas (segmento de dados), sua pilha de chamadas de função (stack) e a área de alocação dinâmica (heap). Além disso, cada processo possui recursos associados, como arquivos abertos, e um conjunto de registradores da CPU, que representam seu contexto de execução imediato. Os mais importantes são:

  • Contador de Programa (Program Counter - PC): Aponta para o endereço da próxima instrução a ser executada.
  • Ponteiro de Pilha (Stack Pointer - SP): Aponta para o topo da pilha do processo, usada for variáveis locais e chamadas de função.

O SO utiliza o conceito de processo para gerenciar múltiplos programas em execução de forma concorrente, mesmo em uma máquina com uma única CPU. Essa técnica é chamada de multiprogramação. O SO alterna a execução entre os vários processos na memória, criando a ilusão de que eles estão rodando em paralelo, o que é conhecido como pseudoparalelismo.

2.2. Estados de um Processo e Transições

Durante seu ciclo de vida, um processo transita por diferentes estados. O modelo mais simples e fundamental inclui três estados básicos:

  • Executando (Running): O processo está, neste exato momento, utilizando a CPU para executar suas instruções. Em um sistema com um único núcleo de processamento, apenas um processo pode estar neste estado a qualquer instante.
  • Pronto (Ready): O processo tem tudo o que precisa para ser executado e está apto a rodar, mas está aguardando que o sistema operacional o selecione e lhe conceda tempo de CPU. Processos neste estado são mantidos em uma estrutura de dados chamada "fila de prontos".
  • Bloqueado (Blocked/Waiting): O processo não pode continuar sua execução, mesmo que a CPU estivesse disponível para ele. Isso ocorre porque ele está esperando por algum evento externo para acontecer. Exemplos comuns incluem esperar pela conclusão de uma operação de entrada/saída (E/S), esperar por um sinal de outro processo, ou esperar que um recurso (como um lock) seja liberado.
[Imagem do diagrama de 3 estados de processo (Executando, Pronto, Bloqueado)]

As transições que governam o ciclo de vida de um processo são:

  1. Executando -> Bloqueado: Ocorre quando o processo inicia uma operação que não pode ser concluída imediatamente (ex: leitura de disco).
  2. Pronto -> Executando: Ocorre quando o escalonador do SO seleciona este processo da fila de prontos para utilizar a CPU.
  3. Executando -> Pronto: Ocorre em sistemas preemptivos. O "quantum" (fatia de tempo) do processo expira, e o escalonador força sua saída da CPU.
  4. Bloqueado -> Pronto: Ocorre quando o evento pelo qual o processo estava esperando finalmente acontece (ex: a leitura de dados foi concluída).

2.3. Implementação de Processos: Tabela de Processos e BCP

Para gerenciar todos os processos ativos, o sistema operacional mantém uma estrutura de dados central, chamada Bloco de Controle de Processo (BCP). O SO mantém uma Tabela de Processos que, essencialmente, é uma lista de ponteiros para os BCPs de todos os processos existentes.

O BCP é a "carteira de identidade" de um processo. Ele armazena todas as informações vitais que o SO precisa para gerenciar aquele processo, incluindo:

  • Contexto de Hardware: Uma cópia de todos os registradores da CPU (PC, SP, etc.) no momento em que o processo foi interrompido.
  • Estado do Processo: O estado atual (Executando, Pronto ou Bloqueado).
  • Identificador do Processo (PID): Um número único que identifica o processo.
  • Informações de Escalonamento: A prioridade do processo, o quantum, etc.
  • Gerenciamento de Memória: Ponteiros para as tabelas de páginas que definem o espaço de endereçamento.
  • Informações de Arquivos e E/S: O diretório de trabalho atual, descritores de arquivos abertos.
  • Informações de Contabilidade: ID do usuário, ID do grupo, etc.

2.4. Mudança de Contexto

Quando o SO decide alternar a CPU de um processo (P1) para outro (P2), ele realiza uma operação chamada mudança de contexto. Este é um procedimento mecanicamente simples, mas com um custo de desempenho significativo:

  1. O SO salva o contexto de hardware completo do processo P1 dentro do seu BCP na memória.
  2. O SO atualiza o estado de P1 em seu BCP (por exemplo, de Executando para Pronto).
  3. O escalonador é invocado para escolher o próximo processo a ser executado, digamos P2.
  4. O SO carrega o contexto de hardware do processo P2, que estava previamente salvo em seu BCP, para os registradores físicos da CPU.
  5. O SO atualiza o estado de P2 para Executando.
  6. A execução do processo P2 é retomada do ponto onde havia parado.

A mudança de contexto é um overhead, um tempo em que a CPU não está executando trabalho útil para nenhuma aplicação, apenas trabalho administrativo para o SO.

2.5. Escalonamento de Processos: Conceitos e Objetivos

O escalonador (scheduler) é o coração da gerência de processos. É o algoritmo que decide qual processo na fila de prontos será o próximo a receber a CPU. Os objetivos do escalonamento são muitas vezes conflitantes e dependem do tipo de sistema:

  • Para todos os sistemas: Justiça (Fairness) e Eficiência (Efficiency).
  • Para sistemas em Batch: Vazão (Throughput) e Tempo de Turnaround.
  • Para sistemas Interativos: Tempo de Resposta.
  • Para sistemas de Tempo Real: Cumprimento de Prazos (Deadlines).

Os algoritmos de escalonamento podem ser de dois tipos principais: Não-preemptivos (um processo só libera a CPU voluntariamente) ou Preemptivos (o SO pode forçar a retirada da CPU). A maioria dos sistemas interativos modernos é preemptiva.

2.6. Algoritmos de Escalonamento para Sistemas em Batch

  • First-Come, First-Served (FCFS ou FIFO): Simplesmente uma fila. É um algoritmo não-preemptivo. Vantagens: simples. Desvantagens: pode ser muito ineficiente e injusto se um processo longo chegar primeiro (o "efeito comboio").
  • Shortest Job First (SJF): O escalonador seleciona o processo com o menor tempo de execução estimado. É não-preemptivo. Vantagens: comprovadamente ótimo para minimizar o tempo médio de espera. Desvantagens: impossível saber de antemão qual será o tempo de execução de um processo e pode levar à inanição (starvation).
  • Shortest Remaining Time Next (SRTN): É a versão preemptiva do SJF. O escalonador sempre escolhe o processo que tem o menor tempo restante de execução. Vantagens: excelente tempo de resposta para processos curtos. Desvantagens: sofre dos mesmos problemas do SJF mais o overhead de mudanças de contexto.

2.7. Algoritmos de Escalonamento para Sistemas Interativos

  • Round-Robin (RR): Um dos algoritmos mais simples, justos e amplamente utilizados. O escalonador mantém uma fila de processos prontos. Cada processo recebe uma pequena fatia de tempo da CPU, chamada quantum. Se o processo não terminar ou se bloquear, ele é movido para o final da fila. O desempenho é muito sensível ao tamanho do quantum.
  • Escalonamento por Prioridades: Cada processo tem uma prioridade. O escalonador sempre escolhe o processo pronto com a maior prioridade. Pode ser preemptivo. O principal problema é a inanição de processos de baixa prioridade. Uma solução comum é o envelhecimento (aging), onde a prioridade aumenta com o tempo de espera.
  • Múltiplas Filas (Multiple Queues): O sistema mantém várias filas de prontos, geralmente associadas a diferentes classes de prioridade (ex: uma fila para processos interativos com RR, e outra para processos em batch com FCFS). É um sistema muito flexível.

2.8. Algoritmos de Escalonamento para Sistemas de Tempo Real

O escalonamento em sistemas de tempo real é totalmente dominado pela necessidade de cumprir prazos (deadlines). Atrasar uma resposta é considerado uma falha.

  • Estático: As decisões de escalonamento são feitas offline, antes de o sistema rodar. Só é viável se todas as tarefas e seus prazos forem conhecidos de antemão.
  • Dinâmico: As decisões são tomadas em tempo de execução (online).

Algoritmos comuns incluem Rate Monotonic Scheduling (RMS) e Earliest Deadline First (EDF).

Capítulo 3: Threads

3.1. Do Processo Pesado à Thread Leve

Um processo tradicional é considerado uma unidade "pesada" de execução, pois agrupa duas responsabilidades distintas: o agrupamento de Recursos (espaço de endereçamento, arquivos abertos, etc.) e o fluxo de Execução (contador de programa, registradores, pilha).

Em muitas aplicações modernas, como um navegador web, é útil ter múltiplos fluxos de execução que operam sobre o mesmo conjunto de recursos. Criar um processo separado para cada tarefa seria ineficiente. A solução é a thread, também conhecida como processo leve (LWP).

[Imagem mostrando a diferença entre três processos e um processo com três threads]

3.2. O Modelo de Threads

Vamos detalhar o que é compartilhado e o que é privado:

Itens por Processo (Compartilhados entre todas as suas Threads):

  • Espaço de endereçamento (segmentos de código, dados e heap)
  • Variáveis globais
  • Arquivos abertos
  • Processos filhos
  • Alarmes e sinais

Itens por Thread (Privados e únicos para cada Thread):

  • Contador de Programa (PC)
  • Registradores
  • Pilha (Stack)
  • Estado (Executando, Pronto, Bloqueado)

Como todas as threads de um processo compartilham o mesmo espaço de memória, a comunicação entre elas é trivial e extremamente rápida. No entanto, como não há proteção de memória entre as threads, uma pode facilmente corromper os dados de outra. Isso torna a sincronização um desafio central.

3.3. Benefícios e Razões para usar Threads

  • Capacidade de Resposta: Em uma aplicação com interface gráfica, a UI pode ser mantida por uma thread, que permanece responsiva, enquanto outras threads executam tarefas longas em segundo plano.
  • Compartilhamento de Recursos: A cooperação entre threads é muito mais eficiente do que a comunicação entre processos, pois elas compartilham memória e recursos por padrão.
  • Economia: Criar uma nova thread e realizar uma mudança de contexto entre threads do mesmo processo é muito mais "barato" (rápido) do que com processos.
  • Utilização de Arquiteturas Multiprocessador: Threads oferecem um caminho natural para o paralelismo real em sistemas multicore.

3.4. Threads em Arquiteturas Multicore e o Paralelismo Real

É crucial entender a diferença entre concorrência e paralelismo:

  • Concorrência (Pseudoparalelismo): Em uma CPU de núcleo único, o SO alterna rapidamente entre as threads, dando a ilusão de que elas estão executando simultaneamente.
  • Paralelismo Real: Em uma CPU multicore, o SO pode despachar múltiplas threads para executar fisicamente ao mesmo tempo, cada uma em um núcleo diferente.

3.5. Tipos de Threads: Modo Usuário vs. Modo Kernel

A implementação de threads pode ser feita no espaço do usuário ou diretamente no núcleo.

  • Threads em Modo Usuário (User-Level Threads): Todo o gerenciamento é feito por uma biblioteca no espaço do usuário, sem o conhecimento do núcleo. São extremamente rápidas para criar e trocar de contexto. A principal desvantagem: se uma única thread fizer uma chamada de sistema bloqueante, o processo inteiro pode ser bloqueado.
  • Threads em Modo Kernel (Kernel-Level Threads): O núcleo do SO tem conhecimento direto e gerencia todas as threads. A criação e a troca de contexto são mais lentas, mas robustas a bloqueios e permitem paralelismo real.

3.6. Implementação e Desafios

Hoje em dia, a vasta maioria dos sistemas operacionais de propósito geral (Windows, Linux, macOS, Android e iOS) utiliza um modelo de threads em modo kernel. O grande desafio da programação multithread reside no gerenciamento do acesso concorrente a recursos compartilhados para evitar condições de corrida e deadlocks.

Capítulo 4: Comunicação e Sincronização Entre Processos

4.1. A Necessidade de Comunicação e Cooperação

Em qualquer sistema multiprogramado, os processos raramente operam em total isolamento. Frequentemente, eles precisam se comunicar, cooperar e competir por recursos. Essa interação levanta três questões fundamentais que o SO e o programador precisam resolver:

  • Transferência de Informação: Como um processo pode passar informações para outro?
  • Exclusão Mútua: Como garantir que processos concorrentes não interfiram uns nos outros de maneira destrutiva ao acessar recursos compartilhados?
  • Sequenciamento: Como gerenciar dependências de ordem, onde um processo só pode prosseguir depois que outro atingiu um determinado ponto?

4.2. Condições de Corrida (Race Conditions)

Uma condição de corrida (race condition) é uma situação anômala que ocorre quando dois ou mais processos (ou threads) manipulam um recurso compartilhado e o resultado final da operação depende da ordem particular em que seus acessos são intercalados pelo escalonador. O resultado se torna imprevisível e geralmente incorreto.

O exemplo clássico é o spooler de impressão. O cenário: um recurso compartilhado (uma variável next_free_slot) e dois processos (A e B) querendo imprimir. A sequência catastrófica se desenrola assim:

  1. Processo A lê a variável next_free_slot e armazena o valor (ex: 7) em um registrador interno.
  2. Preempção: O quantum de A expira. O SO salva o estado de A e escalona B.
  3. Processo B lê a mesma variável e segue; pode acabar escrevendo no mesmo slot e provocar sobrescrita.

O resultado é um desastre: o trabalho de impressão do Processo B pode ser perdido. Este tipo de bug é notoriamente difícil de depurar.

4.3. Regiões Críticas e o Requisito de Exclusão Mútua

A parte do código de um programa onde um recurso compartilhado é acessado é chamada de região crítica. A solução é impor a exclusão mútua: garantir que, se um processo está executando em sua região crítica, nenhum outro processo concorrente possa entrar na sua região crítica (relativo ao mesmo recurso) ao mesmo tempo.

Uma boa solução deve satisfazer quatro condições: Segurança, Flexibilidade, Ausência de Bloqueio Desnecessário e Ausência de Inanição.

4.4. Soluções de Exclusão Mútua com Espera Ocupada (Busy Waiting)

As primeiras tentativas de solução envolviam espera ocupada, onde um processo fica em loop testando uma condição — isso consome ciclos de CPU.

  • Desabilitar Interrupções: Funciona apenas em sistema unicore e é perigoso.
  • Variáveis de Travamento (Lock): Sofre de condição de corrida se a operação de teste+set não for atômica.
  • Instrução TSL (Test and Set Lock): Operação atômica no hardware que resolve a condição no lock, mas mantém espera ocupada.

4.5. Soluções sem Espera Ocupada: Primitivas Sleep e Wakeup

Uma abordagem melhor é bloquear (sleep) processos que não podem prosseguir e acordá-los (wakeup) quando possível. Porém, uma implementação ingênua pode levar ao problema do "wakeup perdido" (ex: produtor acorda consumidor que ainda não dormiu).

4.6. Semáforos: A Solução de Dijkstra

Proposto por Edsger Dijkstra em 1965, o semáforo é uma variável inteira protegida, projetada especificamente para resolver o problema do "wakeup perdido" e outros problemas de sincronização. Um semáforo armazena o número de "wakeups" para uso futuro. Ele suporta duas operações, que o hardware e o SO garantem que sejam atômicas:

Tipos de Semáforos:

  • Mutex (Semáforo Binário): Inicializado com 1. Usado para exclusão mútua. O primeiro processo a executar down passa, definindo o valor para 0. Qualquer outro processo que tente um down será bloqueado até que o primeiro execute up.
  • Semáforo de Contagem: Inicializado com N, usado para controlar o acesso a um pool de N recursos.

Solução do Produtor-Consumidor com Semáforos:

Utiliza três semáforos: mutex, empty, e full. Código (esquemático):

// Código do Produtor:
while(true) {
  produzir_item();
  down(&empty);      // Espera por um slot vazio. Decrementa contador de vazios.
  down(&mutex);      // Entra na região crítica para acessar o buffer.
  inserir_item();
  up(&mutex);        // Sai da região crítica.
  up(&full);         // Sinaliza que um slot cheio foi adicionado. Acorda um consumidor.
}

// Código do Consumidor:
while(true) {
  down(&full);       // Espera por um slot cheio. Decrementa contador de cheios.
  down(&mutex);      // Entra na região crítica para acessar o buffer.
  remover_item();
  up(&mutex);        // Sai da região crítica.
  up(&empty);        // Sinaliza que um slot vazio foi liberado. Acorda um produtor.
  consumir_item();
}

Essa solução resolve tanto exclusão mútua quanto sincronização.

4.7. Monitores: Sincronização de Alto Nível

Um monitor é uma construção de alto nível que encapsula exclusão mútua automaticamente e fornece variáveis de condição (wait(), signal()), presente em linguagens como Java e C#.

4.8. Passagem de Mensagens para Sistemas Distribuídos

Em sistemas distribuídos (sem memória compartilhada), a comunicação é feita por passagem de mensagens com primitivas send(dest,&mensagem) e receive(rem,&mensagem). Introduz desafios como perda de mensagens, retransmissão e autenticação.

4.9. Problemas Clássicos de Sincronização

Problemas teóricos canônicos:

  • O Jantar dos Filósofos: Cinco filósofos, cinco garfos — projetar protocolo sem deadlock nem inanição.
  • Os Leitores e Escritores: Múltiplos leitores podem ler simultaneamente; apenas um escritor por vez.
  • O Barbeiro Dorminhoco: Modelo cliente-servidor com espera limitada (cadeiras de sala de espera).
