Guia de Estudos Completo: Sistemas Operacionais

Um guia detalhado e aprofundado com base nos materiais de aula para sua prova.

Esta seção cobre os conceitos fundamentais de um SO, suas funções, histórico e as diferentes arquiteturas existentes.

1.1 O que é um Sistema Operacional (SO)?

O SO é uma camada de software que atua como intermediário entre o hardware e os aplicativos, simplificando e gerenciando o acesso aos recursos do computador.

Ele desempenha duas funções principais:

  • Máquina Estendida (Abstração de Recursos): O SO esconde a complexidade do hardware (registradores, interrupções, controladores de disco) e fornece uma interface mais simples e limpa (abstrações) para os programadores e aplicativos. Por exemplo, a complexidade de ler um bloco de um SSD é abstraída para uma simples chamada de sistema como `read()`.
  • Gerenciador de Recursos: O SO gerencia todos os recursos do sistema (CPU, memória, E/S) de forma ordenada e controlada, decidindo qual processo usa qual recurso, por quanto tempo e quando. Isso é crucial em sistemas multiusuário e multitarefa para garantir a justiça e a eficiência.

1.2 Tipos e Estruturas de SO

Sistemas operacionais podem ser classificados por sua arquitetura interna:

  • Sistemas Monolíticos: Todo o SO (gerenciamento de processos, memória, arquivos, drivers) roda como um único programa grande no modo kernel. É rápido, mas uma falha em um driver pode derrubar todo o sistema. Ex: Linux, Windows (em grande parte).
  • Sistemas em Camadas: O SO é organizado em uma hierarquia de camadas, onde cada camada usa apenas os serviços da camada imediatamente inferior. Ajuda na organização, mas pode ser ineficiente.
  • Micronúcleo (Microkernel): Apenas as funções mais essenciais do SO rodam no modo kernel (comunicação entre processos, gerenciamento básico de memória). O restante (drivers, sistema de arquivos) roda como processos de usuário. É mais modular e seguro, mas a comunicação entre os processos pode ser um gargalo de desempenho. Ex: Minix 3.
  • Sistemas Híbridos: A maioria dos SOs modernos, como Windows e macOS, combina elementos das arquiteturas monolítica e de micronúcleo para equilibrar desempenho e modularidade.

1.3 Chamadas de Sistema (System Calls)

É a interface que os programas de usuário utilizam para solicitar serviços do núcleo do SO. Como os aplicativos rodam em modo usuário (com privilégios limitados), eles não podem acessar o hardware diretamente. Para realizar operações como ler um arquivo, criar um novo processo ou enviar um pacote de rede, eles precisam fazer uma "chamada de sistema". Esse mecanismo causa um *trap* (uma interrupção de software) que transfere o controle da CPU para o kernel. O kernel então executa a operação solicitada em modo privilegiado e retorna o resultado ao processo do usuário, que pode assim continuar sua execução.

Um processo é a abstração central de um programa em execução. O SO é responsável por criar, destruir e gerenciar todos os processos.

2.1 Modelo e Estados de um Processo

  • Processo: Instância de um programa em execução. É uma entidade dinâmica que possui um espaço de endereçamento, código, dados, pilha e recursos.
  • Estados de um Processo:
    1. Executando (Running): O processo está efetivamente usando a CPU para executar suas instruções.
    2. Pronto (Ready): O processo tem tudo o que precisa para executar, mas está na fila de prontos, aguardando a CPU ser alocada a ele pelo escalonador.
    3. Bloqueado (Blocked): O processo não pode continuar sua execução, pois está esperando por um evento externo. Exemplos: esperando o término de uma leitura de disco, esperando por uma entrada do teclado ou esperando que outro processo libere um recurso. Ele não consumirá CPU enquanto estiver neste estado.

2.2 Bloco de Controle de Processo (BCP)

O BCP (PCB - Process Control Block) é a "identidade" de um processo para o sistema operacional. É uma estrutura de dados fundamental que armazena todo o contexto necessário para gerenciar um processo. Quando o SO realiza uma troca de contexto, ele salva o estado atual da CPU no BCP do processo que está saindo e carrega o estado do próximo processo a partir do seu BCP.

Campos típicos de um BCP incluem:

  • Gerenciamento de Processo: Estado atual, ID do processo (PID), ID do processo pai, prioridade de escalonamento.
  • Gerenciamento de Memória: Ponteiros para os segmentos de código, dados e pilha.
  • Gerenciamento de Arquivos: Lista de arquivos abertos.
  • Contexto de Hardware: Conteúdo dos registradores da CPU, incluindo o contador de programa.

Threads, ou "processos leves", permitem que um único processo execute múltiplas tarefas concorrentemente, oferecendo um modelo mais eficiente para o paralelismo em muitas aplicações.

3.1 O Modelo de Threads

Uma thread é uma unidade de execução dentro de um processo. Múltiplas threads dentro de um mesmo processo compartilham o mesmo espaço de endereçamento e recursos (como arquivos abertos), mas cada uma tem seu próprio contador de programa, registradores e pilha.

3.2 Benefícios do Multithreading

  • Responsividade: Em uma aplicação com interface gráfica, uma thread pode cuidar da interação com o usuário enquanto outra executa uma tarefa demorada em segundo plano. Ex: Um editor de texto salvando um arquivo em uma thread separada sem travar a interface.
  • Compartilhamento de Recursos: Facilita a comunicação e a colaboração entre partes de um mesmo programa, pois todas as threads acessam as mesmas variáveis globais e o mesmo heap de memória.
  • Economia: É muito mais "barato" (rápido e com menos overhead) para o SO criar e gerenciar threads do que processos. A troca de contexto entre threads de um mesmo processo também é mais eficiente.
  • Escalabilidade: Essencial para aproveitar arquiteturas multicore. O SO pode escalonar diferentes threads do mesmo processo para serem executadas verdadeiramente em paralelo em diferentes núcleos de CPU.

3.3 Implementação de Threads: Modo Usuário vs. Modo Kernel

  • Threads em Espaço de Usuário:
    • O kernel não "enxerga" as threads; elas são gerenciadas por uma biblioteca de runtime no espaço do processo.
    • Vantagens: Criação e troca de contexto extremamente rápidas, sem a necessidade de chamadas de sistema.
    • Desvantagem principal: Se uma thread faz uma chamada de sistema bloqueante (ex: `read()`), o kernel bloqueia o processo inteiro, parando todas as outras threads.
  • Threads em Espaço de Kernel:
    • O kernel gerencia as threads diretamente, mantendo uma tabela de threads para todo o sistema.
    • Vantagens: Se uma thread bloqueia, o kernel pode escalonar outra thread (do mesmo processo ou de outro) para executar. É o modelo padrão nos SOs modernos (Windows, Linux, macOS).
    • Desvantagem: A criação, destruição e troca de contexto são mais lentas por requererem chamadas ao sistema.

O escalonamento decide qual processo ou thread na fila de prontos deve receber a CPU. A escolha do algoritmo impacta diretamente o desempenho, a responsividade e a justiça do sistema.

4.1 Escalonamento para Sistemas em Batch

O objetivo principal é a vazão (throughput) e a utilização da CPU, não a interatividade.

  • FCFS (First-Come, First-Served): Simples e justo, mas pode ser muito ineficiente. Processos I/O-bound podem ficar presos atrás de um processo CPU-bound, deixando os dispositivos de E/S ociosos.
  • SJF (Shortest Job First): Provadamente ótimo para o tempo médio de retorno, mas requer conhecimento do futuro (o tempo de execução de cada job), o que é impossível na prática. É usado como uma base teórica de comparação.

4.2 Escalonamento para Sistemas Interativos

O objetivo principal é um bom tempo de resposta para o usuário.

  • Round-Robin: O algoritmo de escalonamento mais fundamental para sistemas interativos. Cada processo recebe um *quantum* de tempo. Se o processo ainda estiver rodando ao final do seu quantum, ele sofre preempção e é movido para o fim da fila de prontos.
  • Prioridades: Cada processo recebe uma prioridade. O escalonador sempre escolhe o processo pronto de maior prioridade. Prioridades podem ser estáticas ou dinâmicas. O perigo é a inanição (*starvation*) de processos de baixa prioridade.
  • Múltiplas Filas: Separa os processos em classes de prioridade (ex: interativos, batch), cada uma com sua própria política de escalonamento (ex: Round-Robin para interativos, FCFS para batch).
  • Loteria (Lottery Scheduling): Um algoritmo probabilístico. Processos recebem "bilhetes de loteria" para recursos (como a CPU). Processos mais importantes recebem mais bilhetes. Quando uma decisão de escalonamento precisa ser feita, um bilhete é sorteado aleatoriamente, e o processo que o possui ganha o recurso.

4.3 Escalonamento em Tempo Real (Real-Time)

O critério mais importante é a previsibilidade e o cumprimento de prazos.

  • Hard Real-Time: Prazos absolutos que não podem ser perdidos. A falha em cumprir um prazo é uma falha catastrófica do sistema (ex: controle de voo de um avião, sistema de freios de um carro).
  • Soft Real-Time: Prazos importantes, mas que podem ser ocasionalmente perdidos sem consequências drásticas (ex: streaming de vídeo, onde perder um quadro não é o fim do mundo).

Esta seção aborda os desafios da execução concorrente e os mecanismos utilizados para garantir que os processos cooperem de forma correta e segura.

5.1 O Problema da Região Crítica e Condições de Corrida

Uma **Condição de Corrida (Race Condition)** ocorre quando múltiplos processos acessam e manipulam dados compartilhados concorrentemente, e o resultado final depende da ordem em que os acessos acontecem. A parte do programa onde o dado compartilhado é acessado é chamada de **Região Crítica**.

A solução é a **Exclusão Mútua**, que garante que apenas um processo esteja dentro de sua região crítica por vez.

5.2 Soluções para Exclusão Mútua

  • Espera Ocupada (Busy Waiting): Soluções como `lock variables` e `strict alternation` são ineficientes pois desperdiçam ciclos de CPU. A **Instrução TSL (Test and Set Lock)** é uma solução baseada em hardware que resolve o problema de atomicidade, mas ainda mantém a espera ocupada.
  • Primitivas Sleep e Wakeup: Uma melhoria sobre a espera ocupada. O processo que precisa esperar é colocado em estado de **bloqueado** (`sleep`) e não consome CPU. Ele é acordado (`wakeup`) por outro processo quando pode prosseguir. Sofre do problema do **"wakeup perdido"**, que pode levar a deadlocks.
  • Semáforos: A solução clássica de Dijkstra. Um semáforo é um contador que suporta duas operações atômicas:
    • `down()` (ou `P` ou `wait`): Se o contador for maior que zero, decrementa. Se for zero, o processo bloqueia.
    • `up()` (ou `V` ou `signal`): Incrementa o contador. Se houver processos bloqueados, acorda um deles.
    O semáforo **Mutex** é um semáforo binário usado para garantir exclusão mútua.
  • Monitores: Uma abstração de mais alto nível que simplifica a programação concorrente. O compilador garante a exclusão mútua em seus métodos. Processos esperam por eventos usando **variáveis de condição** (`wait` e `signal`).
  • Passagem de Mensagens: Processos comunicam-se enviando mensagens uns aos outros, ideal para sistemas distribuídos, usando primitivas como `send` e `receive`.

5.3 Problemas Clássicos de Sincronização

São problemas teóricos usados para testar e demonstrar a eficácia dos mecanismos de sincronização.

  • Produtor-Consumidor: Coordena o acesso a um buffer compartilhado, evitando que o produtor insira em um buffer cheio e o consumidor retire de um buffer vazio.
  • Jantar dos Filósofos: Ilustra o problema do *deadlock*, onde um conjunto de processos fica permanentemente bloqueado, cada um esperando por um recurso que está sendo mantido por outro processo no mesmo conjunto.
  • Leitores e Escritores: Gerencia o acesso a uma base de dados. Múltiplos leitores podem acessar simultaneamente, mas um escritor deve ter acesso exclusivo.
  • Barbeiro Adormecido: Modela problemas de cliente-servidor. O barbeiro (servidor) dorme se não há clientes (requisições). O cliente deve acordar o barbeiro. Se chegam mais clientes enquanto o barbeiro está ocupado, eles esperam (se houver cadeiras/buffers) ou vão embora.

Um processo A e um processo B, esses processos queriam fazer suas impressões, o processo A vai lá e verifica qual é o próximo slot livre na fila de impressão, coloca o seu arquivo para imprimir, contudo acaba o tempo de processamento ali daquele processo e o processo B vai ser executado. E a gente viu em termos práticos que um processo vai acabar sobrescrevendo ali o arquivo do outro. E aí não é uma situação adequada. E a gente viu que para resolver esse tipo de problema aí, a gente adota alguns critérios, né? Como a exclusão mútua, por exemplo, e ela tem que obedecer aí alguns critérios. A gente trabalhou com a ideia de espera ocupada. Qual que é o problema da espera ocupada? O gasto de tempo de CPU testando se eu posso entrar ou não ali na minha região crítica. Então a gente viu que em termos práticos isso não é uma boa solução.

Parte 2 (SO - 11-09 - 2.m4a)

Ainda falando sobre exclusão mútua, né? Então a gente saiu ali de espera ocupada, primitivas sleep/wakeup e chega aí numa solução de exclusão mútua baseada em semáforos. A ideia é que, a partir do momento que eu lanço uma primitiva, seja de sleep ou seja de wakeup, eu lanço um bit correspondente a essa primitiva. E aí no momento da execução dos processos, eu faço a verificação tanto da primitiva quanto do bit. Se esse bit é um bit válido, é um bit 1, significa que há uma correspondência ali daquela primitiva. Se aquele bit é um bit zero, por exemplo, significa que não há uma correspondência ali daquela primitiva e o sistema vai ter que ou colocar o processo para dormir ou vai ter que acordar o processo de acordo com essa informação aí que ele tá monitorando. Basicamente é uma ideia de que é assim, para cada primitiva sleep, eu vou ter que ter uma primitiva wakeup. Então eu não posso ter duas primitivas sleeps para um wakeup. Essas primitivas sempre acontecem aos pares. Então se uma dessas primitivas foi perdida, significa que eu preciso colocar uma delas aí no lugar, seja para o processo que tá acordado, ou seja para o processo aí que tá dormindo.

  • Entenda os Problemas Antes das Soluções: Antes de decorar o que é um semáforo, entenda profundamente por que ele é necessário. Qual o problema da **condição de corrida**? Por que a **espera ocupada** é ruim? Por que a primitiva `sleep` sozinha não resolve?

  • Pratique com os Problemas Clássicos: Os problemas do **Produtor-Consumidor** e do **Jantar dos Filósofos** não são apenas curiosidades. Eles são excelentes ferramentas para solidificar seu entendimento sobre **exclusão mútua, deadlock e starvation**.
